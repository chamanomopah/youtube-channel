# Formatar Comando Slash para Padrão SDLC

Criar uma versão atualizada (v2) de qualquer comando slash existente para seguir o padrão estrutural SDLC (Software Development Life Cycle), mantendo simples e direto ao ponto com a funcionalidade original. Use THINK HARD para seguir as `Instructions` para transformar o `arquivo_alvo` usando a estrutura SDLC. Siga a seção `Relatório` para relatar corretamente os resultados da transformação, criando um arquivo atualizado (v2) e deixando o arquivo citado intacto.

## Arquivos Relevantes

`arquivo_alvo`

## Variáveis

arquivo_alvo: $ARGUMENTS

## Instruções

- IMPORTANTE: não modifique o `arquivo_alvo`, crie uma versão atualizada dele (v2)

### 1. Análise Inicial
- Leia o arquivo `arquivo_alvo` para entender seu propósito atual
- Identifique o tipo principal do comando (feature, bug, chore, implementação, documentação, test.)

### 2. Análise Estrutural do Arquivo Alvo
- Extraia o título/descrição principal do comando
- Identifique variáveis existentes (se houver)
- Liste as instruções/procedimentos atuais
- Identifique o formato/template atual (se houver)
- Determine seções de validação ou relatório

### 3. Adaptação para Formato SDLC-HLP
- Mantenha a funcionalidade original do comando
- Adapte para a estrutura SDLC-HLP usando análise contextual do próprio arquivo
- Preserve instruções específicas do comando
- Adicione seções modulares que façam sentido para o contexto

### 4. Análise Contextual e Definição da Estrutura
- Realize análise de contexto do comando (complexidade, interatividade, dependências, etc.)
- Determine os blocos construtores necessários com base na análise
- Defina a estrutura modular do novo arquivo ANTES de criar a descrição principal
- Organize os blocos em ordem lógica de execução

### 5. Criação do Novo Arquivo
- Crie um novo arquivo atualizado `arquivo_alvo` com o formato adaptativo
- Crie a descrição principal CONTEXTUAL que comunica valor real
- Mantenha toda a funcionalidade original
- Adicione estrutura SDLC-HLP consistente onde aplicável
- Garanta clareza e executabilidade

## Formato Adaptativo SDLC-HLP (High Level Prompting)

### 1. Análise Contextual PRIMEIRO
Em vez de categorização rígida, use análise de contexto para determinar a estrutura ideal:

**Análise de Contexto do Comando:**
- **Complexidade**: [simples/média/complexa]
- **Interatividade**: [automático/interativo/híbrido]
- **Dependências**: [internas/externas/mistas]
- **Validação Necessária**: [sim/não/parcial]
- **Tipo Principal**: [dominante + secundários]

### 2. Construção de Estrutura Modular
Em vez de templates fixos, use blocos construtores adaptativos:

**Core Foundation (sempre presente):**
- ## Contexto & Variáveis
- ## Instruções Principais

**Execution Layers (adicione conforme necessidade):**
- ## Preparação & Análise
- ## Passos Executivos
- ## Validação & Qualidade
- ## Entrega & Finalização

**Communication Layers (adicione conforme necessidade):**
- ## Relatório de Progresso
- ## Documentação de Resultados
- ## Próximos Passos

### 3. Crie a Descrição Principal Contextual
A descrição principal deve comunicar valor real e contexto de uso, seguindo estes princípios:

**Responda às perguntas-chave:**
- O que este comando faz de forma clara e direta?
- Quando alguém deveria usar este comando?
- Qual problema específico ele resolve?
- Qual é o resultado esperado?

**Referencie seções com propósito:**
- Cite ``Instruções`` apenas se ajudar na execução passo a passo
- Mencione ``Relatório`` se o comando gera resultados específicos
- Referencie outras seções quando elas agregam contexto essencial

**Evite:**
- Frases genéricas copiadas de outros comandos
- Referências forçadas sem valor informativo
- Descrições que poderiam se aplicar a qualquer comando

```markdown
# <Título que comunica o resultado real>

<Descrição contextual que explica o que o comando faz, quando usar, e qual problema resolve. Referencie seções apenas quando isso realmente ajudar na execução>

## Contexto & Variáveis
<variáveis específicas para este contexto>

## Instruções Principais
<fluxo de trabalho adaptado ao comando>

## [Seções Específicas]
<apenas seções necessárias para este tipo>

## [Resultados se aplicável]
<seções de comunicação quando relevante>
```

### 4. Princípios de Flexibilidade HLP

**Composição sobre Herança:**
Construa estruturas combinando blocos relevantes em vez de herdar templates fixos

**Adaptação Contextual:**
Modifique a estrutura baseada no caso específico, não em categorias pré-definidas

**Referência Natural:**
Cite apenas seções que realmente adicionam valor à compreensão e execução

**Evolução Progressiva:**
Permita que estruturas evoluam com o uso e feedback real

## Processo de Transformação

1. **Extraia as informações essenciais** do arquivo atual
2. **Analise o contexto** do comando para determinar estrutura adequada
3. **Construa a estrutura modular** usando blocos relevantes
4. **Crie a descrição contextual** que comunica valor real
5. **Adapte as instruções** para a estrutura modular mantendo funcionalidade
6. **Garanta consistência** com outros comandos do projeto
7. **Mantenha simplicidade** e executabilidade

## Diretrizes de Adaptação

- **Preserve funcionalidade**: O comando deve continuar fazendo exatamente o que fazia antes
- **Adicione estrutura modular**: Organize o conteúdo de forma clara e adaptativa
- **Seja contextual**: Adapte as seções ao caso específico, não a categorias fixas
- **Mantenha simplicidade**: Não adicione complexidade desnecessária
- **Garanta evolução**: Crie um formato que possa evoluir com o uso

## Validação Final

Após criar a nova versão do arquivo:
- Verifique se toda funcionalidade original foi preservada
- Confirme que as instruções estão claras e executáveis
- Valide se a estrutura modular faz sentido para o contexto
- Teste se o comando funciona conforme esperado
